# Runtime Rules Skill

This skill enforces which SDK components games must use and which patterns are forbidden. Every game generated by this system must comply.

## Mandatory: PyGame Runtime SDK

All games must use the PyGame Runtime SDK (`pygame_sdk`). Specifically:

### Standard Game Loop

Every game must follow the standard PyGame loop pattern:

```python
def main():
    model = GameModel()
    renderer = Renderer(model.config.screen.width, model.config.screen.height)
    renderer.init()
    clock = pygame.time.Clock()

    running = True
    while running:
        for event in pygame.event.get():
            if event.type == pygame.QUIT:
                running = False
            model.handle_event(event)

        model.update()
        renderer.clear()
        model.render(renderer)
        renderer.present()
        clock.tick(model.config.screen.fps)

    pygame.quit()
```

### SDK Entity system

All game objects (player, enemies, obstacles, bullets, collectibles) must be represented as `Entity` instances. Use the `EntityType` constants:

- `EntityType.PLAYER`
- `EntityType.ENEMY`
- `EntityType.BULLET`
- `EntityType.OBSTACLE`
- `EntityType.COLLECTIBLE`

Use `EntityStore` for managing collections of entities when the game has variable counts (shooter, platformer).

### SDK Renderer

All rendering must go through `Renderer`. Use these APIs:

- `renderer.init()` to initialize display.
- `renderer.init_headless()` for Docker/screenshot mode.
- `renderer.clear(color)` to clear screen.
- `renderer.draw_entity(entity)` to render entities as colored rectangles.
- `renderer.draw_text(text, x, y, color)` for text overlays (menus, game over).
- `renderer.draw_overlay(text, x, y, bg_color)` for overlay panels.

Do not call `pygame.draw` or `pygame.Surface.blit` directly. Use the Renderer abstraction.

### SDK Physics engine

All movement and gravity must use SDK physics functions:

- `apply_gravity(entity, gravity)` for gravity-based movement (flappy).
- `apply_velocity(entity)` for direct velocity-based movement.
- `set_velocity(entity, vx, vy)` for setting velocity.
- `clamp_position(entity, min_x, min_y, max_x, max_y)` for boundary enforcement.

### SDK SceneManager

Game state transitions must use `SceneManager`:

- `SceneManager()` initializes at `SceneState.MENU`.
- `sm.transition(SceneState.GAMEPLAY)` / `sm.transition(SceneState.GAME_OVER)` / `sm.transition(SceneState.PAUSED)` for state changes.
- `sm.is_(SceneState.MENU)` / `sm.is_playing()` for state checks.
- `sm.toggle_pause()` for pause/resume.

### SDK ScoreManager

Score tracking must use `ScoreManager`:

- `ScoreManager()` to create.
- `sm.add(points)` to increment.
- `sm.update_best()` to record high score (returns True if new best).
- `sm.reset()` between rounds.

### SDK DifficultyConfig

Difficulty parameters must be loaded from config.yaml and follow the tiered structure:

- At minimum: `easy`, `normal`, `hard` tiers.
- Each tier is a dict of numeric parameters appropriate to the genre.
- The `params(difficulty)` method on the config selects the active tier with a fallback to `normal`.

### SDK Color Styles

Use the provided color constants for consistent UI:

- `TITLE_COLOR` for game titles.
- `SCORE_COLOR` for score display.
- `GAMEOVER_COLOR` for game-over text.
- `HINT_COLOR` for control hints.
- `GRACE_COLOR` for grace-period text.
- `DIFFICULTY_COLORS` dict keyed by `"easy"`, `"normal"`, `"hard"`.

## Forbidden

The following patterns must NEVER appear in generated game code:

- **No custom game loop**: Do not implement a custom game loop outside the standard pattern shown above. Use the standard while-loop with event processing.
- **No direct PyGame draw calls**: Do not call `pygame.draw.rect`, `pygame.draw.circle`, etc. directly. Use `renderer.draw_entity()`.
- **No custom physics**: Do not write your own gravity, velocity, or collision math. Use SDK functions from `physics` and `collision` modules.
- **No bypassing Entity system**: Do not use plain dicts or lists to track game objects. Use `Entity` and `EntityStore`.
- **No external assets**: No image files, font files, sound files, or any non-Python source file except `config.yaml`.
- **No I/O during gameplay**: No `open()`, `os.system()`, `socket`, `http` during gameplay. Config reads happen once at startup.
- **No unsafe code**: No `exec()`, `eval()`, `__import__()`, `subprocess`, `os.system()` in game code.

## Config Over Code

Games must be configurable via `config.yaml`, not hardcoded:

- Screen dimensions, player color, player HP, and all difficulty parameters must live in `config.yaml`.
- The Python code loads config at startup via `load_config("config.yaml")` with a `default_config()` fallback.
- To change game behavior, edit `config.yaml`. Do not edit Python source.

## Standard File Structure

Every generated game should follow this recommended file structure:

```
<game-name>/
  main.py           -- Entry point. Creates model, runs PyGame loop.
  model.py          -- Model class, constructor, reset().
  config.py         -- Config dataclass, load_config(), default_config(), params().
  config.yaml       -- YAML configuration with all tuneable parameters.
  update.py         -- handle_event(), update() methods.
  view.py           -- render() method using Renderer.
  requirements.txt  -- Dependencies: pygame, pyyaml, pygame-sdk path.
```

Additional files are allowed as needed (e.g., `entities.py`, `spawning.py`).

## SDK Import Convention

Generated games import the SDK via path manipulation:

```python
import sys
import os
sys.path.insert(0, os.path.join(os.path.dirname(__file__), "..", "..", "runtime", "pygame-sdk"))
from pygame_sdk import Entity, EntityStore, Renderer, SceneManager, ScoreManager, ...
```
